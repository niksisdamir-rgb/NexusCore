# 1. Record architecture decisions

Date: 2026-04-12

## Status

Accepted

## Context

We need to record the architectural decisions made on this project so that future developers (and ourselves) can understand the rationale behind our choices. This minimizes repeated discussions and provides a history of the project's evolution.

## Decision

We will use Architecture Decision Records, as described by Michael Nygard in [Documenting Architecture Decisions](http://thinkrelevance.com/blog/2011/11/15/documenting-architecture-decisions).

We will store ADRs in the `docs/architecture/decisions` directory. They will have a numbered prefix (e.g., `0001-record-architecture-decisions.md`) to maintain a chronological history.

## Consequences

See Michael Nygard's article, linked above. For a lightweight ADR format, we use the template shown in this document. Whenever a major architectural shift or technology choice is made, it must be documented here.
